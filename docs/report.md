## 一、算法概述

1. 第一阶段：使用**置换-选择排序**（Replacement-Selection Sort）从输入文件生成若干有序的初始归并段
2. 第二阶段：基于**最佳归并树**策略（多路 Huffman 树），通过多趟 k 路平衡归并将所有归并段合并为全局有序输出

**关于代码来源**：`external_sort.cpp` 中所有代码均为本人亲手编写，未通过 AI 工具直接生成任何代码。AI 仅用于代码审查和文档辅助，详见第六节。本人熟知自己编写的每一行代码，并且有能力进行讲解。代码注释内容相对详细，因此对文档中的算法实现说明进行了精简。

## 二、代码构建过程

本项目全程使用 Git 进行版本管理（共 22 次提交），采用渐进式迭代开发。以下按提交顺序梳理构建过程。

### 2.1 第一阶段：置换-选择排序

`31b61f7` — **核心实现**。从内存版原型出发，将其适配为文件 I/O 版本：

- 用 `std::priority_queue` 维护容量为 4096 的最小堆（缓冲区大小 `M = 4096`，独立于归并路数 `k`），元素类型 `pair<size_t, int>`，按 `(段索引, 数值)` 字典序排序
- 段索引机制：新数 < 刚输出的数时标记为下一段（`index + 1`），否则留在当前段（`index`）——即"冻结"语义
- 段索引变大时自动切换到新输出文件，实现段的自然切换
- 每个元素出堆 → 写入当前段文件 → 读入下一元素 → 计算段索引 → 入堆，单次循环完成

`ebf6361` — 将临时文件统一存放在 `save/` 目录下，方便管理。

### 2.2 第二阶段：最佳归并树 + k 路归并

`5ad9969` — **引入 `_data_count_cache`**。这是一个关键设计决策：`GenerateInitialRuns` 阶段在每个归并段关闭时将其数据条数缓存到全局 `unordered_map<文件名, 条数>` 中。后续构建 Huffman 树时，直接从缓存读取段长度，无需重新扫描文件。在只读访问处使用 `at()` 而非 `operator[]`（`3a02ff8` 修复），避免键不存在时静默插入默认值。

`02e7fa4` — **构建 Huffman 优先队列**。将各段长度压入最小堆，为最佳归并树做准备。每次取最小的 k 个段合并，合并结果作为新段重新入堆。

`ea068e0` — **实现 `_Merge` 辅助函数**。每次合并 k 个有序段时，用最小堆维护 k 个输入流的当前最小值，每次弹出最小的一个写入输出，再从对应流补入下一个数。该函数被 Huffman 归并循环反复调用。

`da6bcdf` — 补齐虚拟段、串联合并流程，完成 `KWayMerge` 主逻辑。

## 三、关键设计要点

### 3.1 从内存原型到文件 I/O 的改造

算法核心基于本人在《数据结构》课程中独立实现的内存版外部排序（仓库 [DataStructure](https://github.com/SiriLee/DataStructure.git)，`include/dsc/algorithm/external_sort.h`）。本项目的关键改造：

- **内存 → 文件**：原型的 `vector<int>` 输入/输出替换为 `ifstream`/`ofstream` 流式读写，借助框架提供的 `IntReader`/`IntWriter` 逐数处理
- **段管理**：原型用 `vector<vector<int>>` 在内存中持有所有段，本项目改为文件存储，通过段索引自动切换输出文件
- **最佳归并树**：原型直接按段长度排序选择，本项目引入 `_data_count_cache` 避免重新扫描文件以获取段长度

### 3.2 `_data_count_cache` 设计

预设接口固定了返回类型，导致文件中数据量信息无法有效记录和传递，因此自行设计了一个静态哈希表，用于统计文件对应的数据量，这是连接两个阶段的关键桥梁。设计考量：

- **写入侧**：`GenerateInitialRuns` 和 `_Merge` 均在文件关闭时将数据条数写入缓存
- **读取侧**：`KWayMerge` 构建 Huffman 树时从缓存读取段长度，不会触发额外 I/O
- **安全性**：使用 `at()` 确保只读不写，键缺失时抛出异常而非静默返回 0
- **生命周期**：函数返回前清空缓存，避免跨次调用污染

### 3.3 文件路径约定

代码中确定的文件组织：

| 路径 | 来源 |
|------|------|
| `save/` | 临时文件目录，`GenerateInitialRuns` 中按需创建 |
| `save/run_<i>.txt` | 初始归并段及中间合并结果，`i` 为全局递增索引，程序主动清理 |
| `save/sorted_output.txt` | 最终输出，由 `KWayMerge` 重命名生成 |

## 四、复杂度分析

- **第一阶段**：每个元素一次堆操作，$O(N \log M)$（$M = 4096$ 为缓冲区大小）
- **第二阶段单趟归并**：每个元素一次堆操作，$O(N \log k)$
- **归并趟数**：置换-选择在随机数据下约产生 $R \approx N/(2M)$ 个初始段，Huffman 树归并约需 $\lceil\log_k R\rceil$ 趟

**总体时间复杂度**：$O(N \log M \cdot \log_k R)$。以 $M=4096$、$N=10^7$ 为例，$R \approx 1222$。k=64 时约需 2 趟归并，k=4 时约需 5 趟。

**空间复杂度**：$O(M + R)$，实际内存开销远小于数据总量（$M, R \ll N$）。
