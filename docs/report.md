## 一、算法概述

外部排序用于处理超出内存容量的海量数据。本项目实现了一个完整的二阶段外部排序系统，两个阶段统一由参数 **k** 控制（第一阶段：内存缓冲区最多容纳 k 个元素；第二阶段：k 路归并的路数上限）：

1. **第一阶段**：使用**置换-选择排序**（Replacement-Selection Sort）从输入文件生成若干有序的初始归并段（run）
2. **第二阶段**：基于**最佳归并树**策略（多路 Huffman 树），通过多趟 k 路平衡归并将所有归并段合并为全局有序输出

算法原型来自本人在《数据结构》课程中独立实现的内存版外部排序（仓库 [DataStructure](https://github.com/SiriLee/DataStructure.git)，`include/dsc/algorithm/external_sort.h`），本项目将其适配为文件 I/O 版本并进行了多项改进。

**关于代码来源**：`external_sort.cpp` 中所有代码均为本人亲手编写，未通过 AI 工具直接生成任何代码。AI 仅用于代码审查和文档辅助，详见第七节。

## 二、代码构建过程

本项目全程使用 Git 进行版本管理（共 22 次提交），采用渐进式迭代开发。以下按提交顺序梳理构建过程。

### 2.1 第一阶段：置换-选择排序

`31b61f7` — **核心实现**。从内存版原型出发，将其适配为文件 I/O 版本：

- 用 `std::priority_queue` 维护容量为 k 的最小堆，元素类型 `pair<size_t, int>`，按 `(段索引, 数值)` 字典序排序
- 段索引机制：新数 < 刚输出的数时标记为下一段（`index + 1`），否则留在当前段（`index`）——即"冻结"语义
- 段索引变大时自动切换到新输出文件，实现段的自然切换
- 每个元素出堆 → 写入当前段文件 → 读入下一元素 → 计算段索引 → 入堆，单次循环完成

`ebf6361` — 将临时文件统一存放在 `save/` 目录下，方便管理。

### 2.2 第二阶段：最佳归并树 + k 路归并

`5ad9969` — **引入 `_data_count_cache`**。这是一个关键设计决策：`GenerateInitialRuns` 阶段在每个归并段关闭时将其数据条数缓存到全局 `unordered_map<文件名, 条数>` 中。后续构建 Huffman 树时，直接从缓存读取段长度，无需重新扫描文件。在只读访问处使用 `at()` 而非 `operator[]`（`3a02ff8` 修复），避免键不存在时静默插入默认值。

`ea068e0` — **实现 `_Merge` 辅助函数**。每次合并 k 个有序段时，用最小堆维护 k 个输入流的当前最小值，每次弹出最小的一个写入输出，再从对应流补入下一个数。该函数被 `KWayMerge` 反复调用。

`02e7fa4` — **构建 Huffman 优先队列**。将各段长度压入最小堆，每次取最小的 k 个段交给 `_Merge` 合并，合并结果作为新段重新入堆。这是最佳归并树的核心。

`da6bcdf` — 补齐虚拟段、串联合并流程，完成 `KWayMerge` 主逻辑。

## 三、关键设计要点

### 3.1 全手写代码

本项目的特殊性在于：算法核心并非从零构思，而是在已有内存版原型的基础上进行适配和增强。关键改造点包括：

- **内存 → 文件**：原型的 `vector<int>` 输入/输出替换为 `ifstream`/`ofstream` 流式读写，借助框架提供的 `IntReader`/`IntWriter` 逐数处理
- **段管理**：原型用 `vector<vector<int>>` 在内存中持有所有段，本项目改为文件存储，通过段索引自动切换输出文件
- **最佳归并树**：原型直接按段长度排序选择，本项目引入 `_data_count_cache` 避免重新扫描文件以获取段长度

### 3.2 `_data_count_cache` 设计

这是连接两个阶段的关键桥梁。设计考量：

- **写入侧**：`GenerateInitialRuns` 和 `_Merge` 均在文件关闭时将数据条数写入缓存
- **读取侧**：`KWayMerge` 构建 Huffman 树时从缓存读取段长度，不会触发额外 I/O
- **安全性**：使用 `at()` 确保只读不写，键缺失时抛出异常而非静默返回 0
- **生命周期**：函数返回前清空缓存，避免跨次调用污染

### 3.3 文件路径约定

代码中确定的文件组织：

| 路径 | 来源 |
|------|------|
| `save/` | 临时文件目录，`GenerateInitialRuns` 中按需创建 |
| `save/run_<i>.txt` | 初始归并段及中间合并结果，`i` 为全局递增索引 |
| `save/sorted_output.txt` | 最终输出，由 `KWayMerge` 重命名生成 |
| `benchmark_big.input` | 测试输入，由 `main.cpp` 生成（非本文件管理） |

### 3.4 错误处理策略

在 AI 审查后，针对 `external_sort.cpp` 可控制的范围内增加了防御性处理：

- 目录不存在时自动创建
- 文件打开后检查 `is_open()`，失败则安全返回
- `_Merge` 中检查 `IntWriter` 返回值，写入失败时关闭流并返回空串
- `k < 2` 时直接返回（涵盖 `k=1` 除零和 `k≤0` 非法输入）
- 最终结果通过 `std::filesystem::rename` 移动到输出路径，中间文件在返回前全部清理

## 四、复杂度分析

- **第一阶段**：每个元素一次堆操作，$O(N \log k)$
- **第二阶段单趟归并**：每个元素一次堆操作，$O(N \log k)$
- **归并趟数**：置换-选择在随机数据下约产生 $R \approx N/(2k)$ 个初始段，Huffman 树归并约需 $\lceil\log_k R\rceil$ 趟

**总体时间复杂度**：$O(N \log k \cdot \log_k R)$。以 $k=64$、$N=10^7$ 为例，$R \approx 78125$，约需 3 趟归并。

**空间复杂度**：$O(k + R)$，实际内存开销远小于数据总量（$k \ll N, R \ll N$）。

## 六、正确性验证

测试框架从三个维度验证算法正确性（均在 `main.cpp` 中实现）：

1. **第一阶段**：遍历所有归并段文件，检查每个文件非空、内部严格升序，且所有文件数据总量与原始输入一致
2. **第二阶段（升序检查）**：遍历最终输出文件，检查全局升序
3. **第二阶段（逐位比对）**：用相同的随机种子重新生成标准答案（内存排序结果），与输出文件逐元素比对

两个阶段均通过以上三项验证，表明算法在正确性上无瑕疵。
